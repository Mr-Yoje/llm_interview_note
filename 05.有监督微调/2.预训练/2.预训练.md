# 2.预训练

\[toc]

### 0. 预训练基础（先建立全局认识）

**预训练（Pretraining）**：在海量无标注文本上做“下一个 token 预测”，让模型学会语言规律、事实知识和推理模式。  
对 Decoder-Only 大模型来说，典型目标函数是自回归语言建模（Causal LM）：

\[
\max \sum_{t=1}^{T}\log P(x_t \mid x_{<t})
\]

可以把它理解成：给定前文，尽可能准确地预测下一个词。

**预训练阶段主要学到什么：**

1.  语言能力：语法、语义、篇章连贯性。
2.  通识知识：事实、概念、常识关系。
3.  泛化能力：在新任务上做 Zero-Shot / Few-Shot 的基础能力。

**预训练阶段不擅长什么：**

-   不保证“按人类指令格式回答”。
-   不保证价值观和安全对齐（这部分通常在 SFT / RLHF 做）。

**一句话区分三个阶段：**

-   预训练：学“知识和能力”  
-   指令微调（SFT）：学“按要求回答”  
-   对齐训练（RLHF/DPO 等）：学“更符合人类偏好”

**预训练的核心产出**是一个“能力强但未充分对齐”的基座模型（Base Model），后续再进入 SFT 和对齐阶段。

**常见预训练目标补充：**

-   **CLM（Causal Language Modeling）**：主流 Decoder-Only 目标，预测下一个 token。
-   **MLM（Masked Language Modeling）**：BERT 系列常见，随机 mask 后重建。
-   **PrefixLM / Span 损失**：部分模型会混合使用，但工业界主流大模型以 CLM 为主。

**损失与指标：**

-   训练时常监控 `cross-entropy loss`。
-   也常看 `perplexity (PPL)`，其与 loss 的关系是 `PPL = exp(loss)`（在同一 tokenization 下可比较）。
-   loss 下降是必要条件，不是充分条件：还要看下游任务和安全评测。

### 1. 为什么要增量预训练？

**预训练学知识**，**指令微调学格式**，**强化学习对齐人类偏好**，所以要想大模型有领域知识，得增量预训练（靠指令微调记知识不靠谱，不是几十w条数据能做到的）。

**再具体一点，增量预训练常见适用场景：**

1.  **补领域知识**：如金融、医疗、法律、代码、企业私域知识。
2.  **补语言能力**：如中文/小语种能力不足，或中英混合能力弱。
3.  **补文体能力**：如学术写作、客服话术、企业 SOP 风格。
4.  **降幻觉率**：通过高密度高质量语料提升事实一致性（不是彻底消除幻觉）。

**什么时候不建议先做增量预训练：**

-   只想让模型“更听话”，优先 SFT / DPO。
-   只有几万到几十万指令数据、几乎没有持续语料，直接做 SFT 性价比更高。
-   算力非常紧张时，先做 LoRA-SFT 验证需求闭环，再决定是否继续预训练。

### 2. 进行增量预训练需要做哪些准备工作？

1.  **选取底座模型**  
    根据任务类型（通用对话 / 代码 / 多语种）、参数规模（7B/14B/32B）和硬件条件做选择。  
    经验上：先用同家族小模型跑通流程，再上大模型，能显著降低试错成本。
2.  **收集数据**  
    来源通常包括：网页语料、百科、代码仓库、论文、问答、行业文档、企业内部文档。  
    规模一般为百亿到万亿 token，不同阶段数据质量要求不同。
3.  **数据清洗与去重**  
    核心是提高信息密度，降低噪音和重复：  
    - 去 HTML 垃圾、模板文本、导航脚注、乱码文本  
    - 去重（文档级/段落级/MinHash）  
    - 过滤低质量样本（超短、符号占比异常、语言识别异常）  
    - 安全过滤（违规内容、隐私信息、敏感数据）
4.  **数据配比设计**  
    不同来源并非“越多越好”，而是要按目标能力配比（如中文、代码、数学、领域语料比例）。  
    需要避免单一来源占比过高导致能力偏科。
5.  **分词器策略**  
    尽量复用底座 tokenizer 保持兼容；若新增词表，要评估 embedding 扩容、OOV 降低和训练稳定性的权衡。
6.  **训练与评测基线**  
    在开始训练前先固化 baseline（loss、PPL、下游任务分数），否则很难判断增量训练是否真的有效。

### 3. 增量预训练所用训练框架？

-   **超大规模训练（多机多卡）**：优先 3D 并行（TP + PP + DP），常见技术栈 Megatron-DeepSpeed / FSDP。  
-   **中等规模训练（少量节点）**：可用 FSDP + 梯度检查点 + 混合精度，兼顾易用性和吞吐。  
-   **少量卡训练（资源受限）**：可用 QLoRA/LoRA 做继续预训练（也叫 CPT with PEFT），以较低显存成本补知识。  

**并行策略简记：**

1.  **数据并行（DP）**：每张卡一份模型，切 batch，最通用。
2.  **张量并行（TP）**：同一层切分到多卡，依赖高速互联（NVLink/IB）。
3.  **流水并行（PP）**：按层切分到多卡，适合超深大模型。
4.  **ZeRO/FSDP**：切分优化器状态、梯度、参数，显著降低单卡显存。

### 4. 增量预训练数据选取思路有哪些？

垂直领域预训练有三种思路：

-   先用大规模通用语料预训练，再用小规模领域语料二次训练
-   直接进行大规模领域语料预训练
-   通用语料比例混合领域语料同时训练

**这三种思路怎么选：**

1.  **通用后领域（二阶段）**：最稳妥，适合“先保通用能力，再补垂域”。
2.  **纯领域预训练**：领域能力提升快，但更容易遗忘通用能力。
3.  **混合训练**：通过配比平衡通用与垂域，工程上最常见。

**常见风险：灾难性遗忘（Catastrophic Forgetting）**

-   现象：领域能力变强，但通用问答、推理、写作能力下降。
-   缓解：保留一定比例通用语料、分阶段降学习率、定期回归评测。

### 5. 增量预训练训练流程是怎么样？

1.  **数据预处理**  
    文本清洗 -> 去重 -> 采样配比 -> 分词 -> 打包为训练样本。  
    长度常设为 2k/4k/8k；长度越长，显存和吞吐压力越大。
2.  **分词器与词表确认**  
    复用底座 tokenizer 时需重点评估中文切分效果和 token 开销。  
    若扩词表，需处理 embedding resize 和初始化策略。
3.  **加载基座模型**  
    继续预训练建议使用 Base 模型而非 Chat 模型，避免已有对齐分布干扰。  
    同时核对权重命名、rope 配置、归一化类型等关键超参。
4.  **设置训练配方（Recipe）**  
    关键项包括：学习率、warmup、batch size、max steps、weight decay、grad clip、混合精度策略。  
    经验上先小步数试跑，确认 loss 正常下降，再拉长训练。
5.  **训练监控与中途评测**  
    在线监控 loss、吞吐、显存、梯度范数、异常值（NaN/Inf）。  
    每隔固定 step 做小规模离线评测，防止“loss 下降但能力退化”。
6.  **Checkpoint 保存与转换**  
    保存策略要考虑故障恢复（如每 N step 保存）和最终部署格式转换（HF、safetensors 等）。
7.  **效果评估与验收**  
    不仅看续写，还要看：领域基准、通用基准、安全基准、事实性和幻觉率。  
    验收通过后再进入 SFT / DPO 阶段。

### 6. 预训练中最关键的超参数（面试高频）

1.  **学习率（LR）**：过大易发散，过小收敛慢；常配合 warmup + cosine decay。
2.  **全局 batch size**：影响训练稳定性和收敛速度；需结合梯度累积控制。
3.  **训练 token 总量**：决定模型“见过多少世界”，与最终能力高度相关。
4.  **上下文长度（context length）**：影响长文能力和显存成本。
5.  **数据质量与配比**：通常比“盲目加数据量”更重要。

### 7. Scaling Law（扩展定律）简述

在一定范围内，模型效果与三者相关：

-   参数规模（Model Size）
-   训练 token 数（Data Size）
-   训练计算量（Compute）

直觉上不是“只堆参数”就行，而是三者要平衡。  
算力固定时，参数太大但 token 不够，会出现“没喂饱”；参数小但 token 过多，会到收益递减区。

### 8. 常见问题与排查

1.  **loss 不降或震荡大**  
    优先检查：学习率过大、数据脏样本、batch 太小、混合精度溢出。
2.  **训练中出现 NaN/Inf**  
    检查梯度裁剪、动态 loss scale、异常样本、初始化和并行通信稳定性。
3.  **领域能力提升但通用能力下降**  
    大概率是遗忘，增加通用语料回灌并做阶段性回归评测。
4.  **中文效果差、token 开销高**  
    先检查 tokenizer 切分，再决定是否扩词表或替换 tokenizer。

### 9. 面试速记（可直接背）

-   预训练本质是用海量无标注数据做下一个 token 预测，学知识与泛化能力。  
-   增量预训练适合补领域知识和语言能力，不是替代 SFT 的“格式训练”。  
-   成败关键在数据质量与配比、训练 recipe 稳定性、以及持续评测闭环。  
-   工程上要同时防三件事：训练不稳定、灾难性遗忘、loss 与真实能力脱钩。
